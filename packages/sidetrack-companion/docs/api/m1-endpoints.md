# Companion HTTP API RFC: M1 Endpoint Groups

## Summary

M1 exposes a localhost-only HTTP API used by the Sidetrack extension to send captured events and tracker mutations into the companion. The companion is the only writer to the vault; every state-changing endpoint writes an audit event.

This RFC gates route implementation. The matching OpenAPI contract lives at `packages/sidetrack-companion/openapi.yaml` and must lint clean before any route lands.

## Consumer/problem

The Chrome MV3 extension needs a short-lived request/response API because service workers can die between captures. Requests must be retry-safe where extension replay is expected, and all inputs are treated as `unknown` until schema-validated.

## Resource design

| Group       | Method | Path                             | Operation ID       | Tags          | Version |
| ----------- | ------ | -------------------------------- | ------------------ | ------------- | ------- |
| Health      | GET    | `/v1/health`                     | `getHealth`        | `health`      | v1      |
| Status      | GET    | `/v1/status`                     | `getStatus`        | `health`      | v1      |
| Events      | POST   | `/v1/events`                     | `appendEvent`      | `events`      | v1      |
| Threads     | POST   | `/v1/threads`                    | `upsertThread`     | `threads`     | v1      |
| Workstreams | POST   | `/v1/workstreams`                | `createWorkstream` | `workstreams` | v1      |
| Workstreams | PATCH  | `/v1/workstreams/{workstreamId}` | `updateWorkstream` | `workstreams` | v1      |
| Queue       | POST   | `/v1/queue`                      | `createQueueItem`  | `queue`       | v1      |
| Reminders   | POST   | `/v1/reminders`                  | `createReminder`   | `reminders`   | v1      |

## Request schema

Schemas are defined in `openapi.yaml` components. Boundary DTOs intentionally preserve M1 vocabulary:

- capture events append to `_BAC/events/<date>.jsonl`
- threads write `_BAC/threads/<bac_id>.json`
- workstreams write `_BAC/workstreams/<bac_id>.json`
- queue items write `_BAC/queue/<bac_id>.json`
- reminders write `_BAC/reminders/<bac_id>.json`

## Success response schema

Every mutation returns:

```json
{
  "data": {
    "bac_id": "01HXEXAMPLE0001",
    "revision": "01HXEXAMPLE0002",
    "requestId": "01HXREQUEST0001"
  }
}
```

Health and status return stable response schemas defined in `openapi.yaml`.

## Error behavior

All non-2xx responses use the shared `Problem` schema from `configs/openapi/openapi.base.yaml`: top-level `type`, `title`, `status`, `code`, `correlationId`, with optional `detail`, `instance`, and `errors[]`.

| Condition                     | Status | Code                    | Notes                           |
| ----------------------------- | -----: | ----------------------- | ------------------------------- |
| Validation failed             |    400 | `VALIDATION_ERROR`      | Field-level `errors[]` included |
| Missing or invalid bridge key |    401 | `AUTHENTICATION_FAILED` | All non-health endpoints        |
| Non-loopback origin           |    403 | `LOOPBACK_ONLY`         | Fail closed                     |
| Unknown resource              |    404 | `NOT_FOUND`             | For future read/update paths    |
| Version conflict              |    409 | `CONFLICT`              | Mutable resources use revisions |
| Vault unavailable             |    503 | `VAULT_UNAVAILABLE`     | Extension queues and retries    |

## Auth/authz

`GET /v1/health` is unauthenticated. Every other route requires `x-bac-bridge-key`, generated by the companion in `_BAC/.config/bridge.key`. The companion rejects non-`127.0.0.1` / `localhost` origins explicitly.

## Idempotency/concurrency

`POST /v1/events` and `POST /v1/queue` require `Idempotency-Key` because the extension replays queued writes after companion downtime. Mutable resources carry a `revision` for optimistic concurrency once update routes are implemented.

## Pagination/filtering/sorting

M1 has no list endpoints on the companion HTTP surface. Future list endpoints use cursor pagination per `standards/01-api-component.md`.

## Observability

- Span names: `companion.health.get`, `companion.status.get`, `companion.events.append`, `companion.threads.upsert`, `companion.workstreams.create`, `companion.workstreams.update`, `companion.queue.create`, `companion.reminders.create`
- Log fields: `requestId`, `route`, `method`, `status`, `durationMs`, `outcome`, `errorCode`, `bac_id`
- Metrics: request count, latency, status class, validation failures, auth failures, vault write failures, idempotency replays
- Audit event: every write logs `requestId`, route, input entity id if present, resulting `bac_id`, revision, timestamp, and outcome to `_BAC/audit/<date>.jsonl`

## Extension model

Routes register through a route registry keyed by stable operation ID. New endpoint groups add a route module plus OpenAPI path/schema entries; existing route dispatch and domain services stay closed to modification.
