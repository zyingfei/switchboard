# Recommendation-System / Data Architecture Review — 2026-07-11

Scope: Sidetrack assessed as what it is under the hood — a single-user, on-device
recommendation system over an event-sourced personal data platform. Produced by a
23-agent review workflow (6 subsystem analysts, adversarial claim verification,
3 re-architecture stances, judged synthesis) against `main` @ `224b102c`.
Companion doc: `2026-07-11-architecture-audit.md` (systems/safety lens).

## The regime (what all judgments are calibrated against)

N=1: no fleet data, labels arrive at human browsing speed (tens/day), cold-start is
permanent-ish. Compute: a Bun process coexisting with live browsing — CPU runaways are
the dominant historical failure mode; memory floor already 2.4-3.6GB. Privacy-first:
nothing leaves the device. Fleet-scale practice does not transfer; several industry-
standard moves are anti-patterns here (see final section).

## Overall assessment (judge)

This is a small-data, single-user, on-device learning-to-rank + graph-inference system whose engineering muscle memory is fleet-scale but whose regime is N=1. The recsys core is correctly conservative where it counts: exact-kNN over a few thousand vault items (no learned retrieval), full-history retrain in seconds, explicit hand labels as gold, a default-OFF clamped online head, and a real offline counterfactual ship gate. That is the right architecture for tens-of-labels-per-day and must be preserved. What is OVER-built for N=1: (a) a LambdaRank listwise objective that is statistically hopeless at one-positive-per-impression; (b) a 50-positive-group ship floor that is a fleet number hard-coded past the env knob (evaluateRecallImpressionShipGateV2 omits minImpressionsWithPositive, falling to DEFAULT=50); (c) an event-store/JSONL DUAL read path (store===null fork) in every hot consumer, with the store default-OFF for a measured-real reason, so the fork buys nothing today and is pure maintenance tax; (d) ~9 on-disk formats with three copies of latestPerAggregateFromLog and an implicit, uncoded derivation DAG. What is UNDER-built and is the true ceiling: (1) POINT-IN-TIME CAPTURE — RecallServedCandidateSnapshot logs perLaneRanks/fusedScore/rerankScore/servedPosition but NOT the extractFeatures vector nor the query-anchored cosine, so the trainer re-derives features at retrain time against a DRIFTED graph (retrain-impressions.ts:416 extractFeatures with merged=[...merged]) and the single most-relevant signal (query cosine, computed at request time) is discarded — serving is effectively lexical+RRF+cross-encoder+hand-tiering; (2) LABEL ECONOMICS — unjudged-but-shown candidates are pushed to scoringRows and DROPPED from training rows (a group is one positive plus explicit rejects), and negative edge feedback has no persisted channel (feedbackOverlay filters only visit_in_topic at serve time; userAssertedRelations is positive-only); (3) NO SCORED OFFLINE EVAL below the ranker path (connections precision runs test-only against synthetic FocusEvalPack; the gate promotes on any epsilon nDCG/MRR delta with no significance test) and no production graph-quality signal; (4) GROWTH WITHOUT LIFECYCLE — the canonical JSONL log (88-92% engagement.interval) has no compaction and the default training read is a whole-log streamFiltered byte-scan every retrain (verified: eventStoreEnabled default-OFF → connectionsMaterializer.ts:4154 streamFiltered fallback); (5) CPU HYGIENE — the impression retrain trains LightGBM INLINE on the drain thread with no cooldown (retrain.ts:890-956), the exact re-embed/rebuild shape behind the historical 100-144% CPU incidents, even though trainGroups.worker.ts and runMaybeRetrainInWorker already exist. Net: the binding constraint is data (coverage + point-in-time truth + label density + measurement), never algorithm capacity. The three stances agree on this diagnosis; they differ only in emphasis (label-loop vs storage-CPU vs learning-objective), which resolves cleanly into one ordered plan because their freeze-safe moves are complementary, not competing.

## Subsystem findings

### label-loop

Rigorous label pipeline but mis-fit to N=1.

**Strengths:**
- Explicit-only labels; reserved-test ship-gate; trainer/gate split parity; exactly-once ledger; default-OFF clamped online head (retrain-impressions.ts:54-105; shipGateV2.ts; onlineHead.ts:57-99).

**Weaknesses (verified):**
- **[high] No-cooldown per-drain retrain, survivorship-biased negatives, and four mediums** (`packages/sidetrack-companion/src/ranker/retrain.ts:890`)
  The throttle (326-372) guards only the legacy path; the impression path (890-956) re-runs the build, trains LightGBM inline, and gates every drain once positives meet the floor with no cooldown (flag ON, materializer:110,4128). Negatives are only explicit rejects among served items (421-431); negatives.ts samplers only in legacy (retrain.ts:708). Mediums: served_position raw, no propensity (feature-schema.ts:99); online head zeroes retrieval features (onlineHead.ts:176-180); reconstruction dedup relies on an unemitted referencesEventId (retrain-impressions.ts:459 vs server.ts:6439); lambdarank weak at one-positive-per-impression (train.ts:731).

**Re-arch opportunities:**
- _freeze-safe, effort M, impact high_ — **Persist point-in-time features at impression time**: Record features into recall.served at serve time or filter context to as-of impression time. Removes leakage and per-drain rebuild cost. Logging/storage only.
- _POST-LIFT, effort S, impact high_ — **Right-size promotion floors for N=1, unify to one env-tunable number**: Thread one floor through the gate (today hard-codes 50), on the full holdout or k-fold. Gate threshold NOT freeze-safe; plumbing unification is.
- _freeze-safe, effort M, impact high_ — **Cooldown/fingerprint dedup on the impression retrain, offload to the worker**: Extend the planRankerRetrain guard to the impression path so no-new-label drains short-circuit, and run the train off-thread via trainGroups.worker. No scoring change.
- _POST-LIFT, effort L, impact high_ — **Break the loop plus position-propensity logging**: Seeded exploratory serve slots plus negatives.ts hard-negative injection plus per-position action-rate logging for later IPW and dropping served_position. Injection/logging freeze-safe; serve/feature changes wait.

**Claims refuted by verification (recorded so they are not re-raised):**
- **Unpromotable gate floor plus point-in-time feature leakage** — Both sub-claims have real code behind them but are overstated in severity for the single-user on-device regime. Sub-claim 1 (gate floor / cold-start lock): Confirmed that evaluateRecallImpressionShipGateV2 at retrain-impressions.ts:753 omits minImpressionsWithPositive, so shipGateV2Decide falls back to DEFAULT_MIN_IMPRESSIONS_WITH_POSITIVE=50 (shipGateV2.ts:94,120). And SIDETRACK_RANKER_IMPRESSION_MIN_GROUPS only lowers the training-run floor (retrain.ts:890 via minRecallImpressionPositiveGroups()), not the gate's threshold. However, this is not a permanent promotion blocker. When reservedTestGroupIdsFor returns null (fewer than 4 groups with at least 2 distinct timestamps), the reserved scoring groups list is empty (line 743-745), so impressionsWithPositiveCount=0 and the gate returns unavailable_insufficient_groups — but this same condition would block a fair gate evaluation regardless of threshold. Once enough groups accumulate (≥4 with temporal spread) the gate's reserved-test impressionsWithPositiveCount will satisfy the 50-group floor naturally, because the reserved test slice already needs to be 20% of a meaningful training set. The claim that it "never promotes at N=1" is trivially true but misleading — it correctly blocks promotion until sufficient data exists for a valid held-out test. Sub-claim 2 (future feature leakage via referenceMs): Confirmed that referenceMsFor (features.ts:842-856) computes the max across ALL events and ALL snapshot edges, and both the served-impression path (line 416) and reconstruction path (line 507) pass merged=[...merged] (the full training history). So early impressions have their recency features evaluated against a referenceMs anchored to the latest event in the full history, not to the impression's own serve time. However, referenceMs is a single constant applied uniformly to ALL candidates within every impression group. LambdaMART trains on within-group relative rankings (pairwise/listwise), not absolute feature magnitudes. A uniform offset to referenceMs shifts all candidates' recency scores by the same multiplicative factor (exp(-delta/30)), which does not change the relative ordering of candidates within any impression group. Therefore, this leakage does not bias the learned ranking model — it only affects the absolute scale of recency features, not which candidate is ranked higher within a group. In the on-device single-user regime with a modest event history window, the practical effect is negligible.

### retrieval-funnel

Isolation: known-good strengths+5-weaknesses base, real rearch added.

**Strengths:**
- Train/serve feature parity is exact (features.ts:1352; learnedRerank.ts:196); recency is data-referenced (features.ts:842); ship gate refuses below 50 positive groups (shipGateV2.ts:96).
- Learned rerank is a background-TTL no-op unless a gate-passed model is active (learnedRerank.ts:221); dense RRF is per-embedder gated (model-registry.ts:151).

**Weaknesses (verified):**
- **[high] served_position is a model input, creating a position-bias feedback loop** (`packages/sidetrack-companion/src/ranker/train.ts:425`)
  served_position is a trained input set from the serve rank at train time, so the model learns top-slot equals relevant and re-consumes it at serve; at N=1 nothing averages the bias out.
- **[medium] Learned reranker rarely engages at N=1 and only re-sorts a cross-encoder list** (`packages/sidetrack-companion/src/recall-v2/learnedRerank.ts:221`)
  Fires only with flag on, model active, ship gate passed, which the 50-positive floor makes rare, so serving is effectively RRF, cross-encoder, then hand-tuned tiering.
- **[medium] anchorId differs between train and serve for query-only recall** (`packages/sidetrack-companion/src/recall-v2/pipeline.ts:1287`)
  No currentUrl: serve anchors on the query while train uses a recall-query prefix, so the pair key misses and serve falls to a by-target lookup; latent skew masked only because neither is a real visit.
- **[medium] Impression training has no negative sampling; unjudged candidates dropped** (`packages/sidetrack-companion/src/ranker/retrain-impressions.ts:421`)
  Only labeled candidates become rows and implicit negatives are zero, so a group is one positive plus rejects and the model never learns to beat shown-but-unclicked items.

**Re-arch opportunities:**
- _freeze-safe, effort M, impact high_ — **Log per-request query-anchored cosine as a first-class feature**: Log the query-to-candidate cosine, already computed at request time, into the served snapshot to revive the dead semantic signal.
- _freeze-safe, effort S, impact high_ — **Drop or debias served_position**: Remove served_position or use it as an inverse-propensity term.
- _POST-LIFT, effort M, impact high_ — **Densify labels and add co-visitation and time features**: Treat unjudged shown candidates as weak negatives; add session co-visitation and time-of-day features.

**Claims refuted by verification (recorded so they are not re-raised):**
- **Pairwise semantic features are structurally dead on this vault** — The claim asserts that "pairwise semantic features" (cosine_similarity and content_vector_cosine) are structurally near-zero at low vector coverage because they read persisted similarity edges rather than live embeddings. Reading the code directly: 1. cosine_similarity (features.ts:1069-1079) reads edge.metadata['cosine'] from persisted visit_resembles_visit edges built by buildSimilarityEdgesByPair (line 1023). The claim is correct that this reads persisted edges — but the critical point is that these edges can come from the lexical Jaccard fallback path (visitSimilarity.ts lines 1067-1103), which produces non-zero cosine scores (reusing the Jaccard score) without any embedding/vector coverage. The lexical path is default-on (lexicalFallbackEnabled() checks the env var, and the code in visitSimilarity.ts line 88 shows it enables by default). So cosine_similarity is NOT structurally near-zero at low vector coverage. 2. content_vector_cosine (features.ts:1396) reads channels.contentVector from the same persisted edge's metadata. This IS genuinely gated on doc embedding coverage: visitSimilarity.ts:700-708 shows it requires leftVectorRef and rightVectorRef to be defined and compatible. With SIDETRACK_PAGE_EVIDENCE_BACKGROUND_EMBEDDING=off (the default, per memory note at 13.6% coverage), content_vector_cosine will be 0 for most pairs. This part of the claim holds for content_vector_cosine specifically. 3. However, the claim's sweeping conclusion that "pairwise semantic features are structurally dead" and "only dense_doc_score carries query relevance" is wrong on two counts: (a) cosine_similarity remains active from the lexical fallback path, producing non-zero values proportional to lexical overlap; (b) bm25_score (from the page_content and timeline_visit recall lanes) also carries query relevance. The claim conflates the two distinct features and incorrectly implies cosine_similarity collapses when embeddings are sparse. The mechanism the claim describes (edge-based reads) is real, but the consequence — near-zero pairwise semantic features at low vector coverage — is refuted by the lexical fallback path that populates cosine_similarity edges independently of embedding availability.

### content-pipeline

Data-quality and dead-lane debt dominate, not CPU: title-only vectors pollute docs_vec unflagged (aggregator false-friends) and clean chunk vectors in documents_chunks_vec are embedded but never queried.

**Strengths:**
- Embedding runs off the main event loop in a forked child sidecar (embedderClient.ts) via setEmbedderOverride (companion.ts:929-930).
- The engagement gate is materialize-time (visitSimilarity.ts:1107), not capture-time; lowering it recovers data.
- recall-v2 is a rebuildable projection over canonical JSON; dedup keyed by embedTextHash (embeddingCache.ts:6-12).

**Weaknesses (verified):**
- **[high] Chunk vectors never queried; content-lane worker never drained; cache rewrites whole file per put** (`packages/sidetrack-companion/src/recall-v2/store/backfill.ts:477`)
  backfillChunkVectors embeds real chunk text into documents_chunks_vec but pipeline.ts uses only docs_vec; drainContentLaneQueue is never called; the cache rewrites the whole file per put.

**Re-arch opportunities:**
- _freeze-safe, effort M, impact high_ — **Tag vectors with content-vs-title provenance so retrieval can down-weight title-only hits**: Add body_indexed/vector_kind to docs_vec, thread through store.queryVector. Logging freeze-safe; serving weight change gated by the freeze.
- _freeze-safe, effort M, impact high_ — **Serve from documents_chunks_vec with max-chunk pooling**: Add queryChunkVector KNN + doc-level pooling, prefer over docs_vec when chunks exist. Query method freeze-safe; flipping serving gated by the freeze.

**Claims refuted by verification (recorded so they are not re-raised):**
- **Title-only vectors pollute docs_vec unflagged; retrieval cannot distinguish content from title-only** — The claim bundles two accurate sub-observations with a false causal link. What is accurate: - ~86% of embedded records (13.6% doc-vector coverage, from MEMORY.md B6 measurement) are embedded with title+pathTokens only because page-text auto-extract is off for most URLs. - backfillVectors (backfill.ts:353-392) copies the full JSON sidecar — both content-derived and title-only vectors — into docs_vec with no provenance flag. - queryVector (sqlite.ts:559-575) returns entityId, canonicalUrl, title, cosineDistance only — no field distinguishes content-derived from title-only embeddings. What is wrong (the load-bearing part of the claim): The claim asserts this is the "source of aggregator false-friends." The actual HN AI-video false-friend incident (fixed in commit 60369f18) was caused by same_repo_or_domain (score 0.65) and same_title_path_tokens (score 0.45) — structural/lexical candidate generators in ranker/candidates.ts, not the semantic vector path. The fix landed in candidates.ts (COARSE_MULTI_TOPIC_DOMAINS guard) and policy.ts (similarity-dominant corroboration gate). The docs_vec / queryVector path feeds generateSemanticQuery in recall-v2/pipeline.ts (the Déjà-vu / related-pages recall surface). That pipeline has no connection to the attribution aggregator; the two pipelines are separate data paths. No code path was found connecting queryVector output to the aggregator grouping logic. The missing-provenance issue in docs_vec is a real quality gap (title-only vectors can pollute recall ranking), but it is not the mechanism behind the aggregator false-friends described in the claim. The severity framing ("source of aggregator false-friends") is unsupported by the code.

### graph-product

The connections graph is a mature, evidence-first data product with genuine strengths: a well-typed edge model (~40 edge kinds in types.ts) that carries provenance (producedBy source + revisionId) and a confidence tier (asserted/observed/inferred), a clean candidate-source generator interface in ranker/candidates.ts, skip-gated topic producers that bound per-drain CPU, and a defense-in-depth aggregator guard landed after the linux-security false-friend. But three N=1-relevant gaps stand out. (1) Provenance-without-lifecycle: inferred similarity edges have no TTL/staleness/re-validation model — a title-only chrome-only resemblance edge persists in the snapshot until its endpoint visit is re-observed via a new timeline event; PAGE_EVIDENCE_EXTRACTED (a content vector arriving later) is a CONTENT_LANE_ONLY handle that does NOT re-embed the visit or recompute its edges, so better evidence never revalidates bad inference. (2) The served similarity path (buildHnswVisitSimilarity) emits cosine-only edges with NO channels metadata, so the aggregator guard's primary discriminator (isChromeOnlySimilarityEdge reading metadata.channels.contentVector) has nothing to read on the actually-served edges — the guard silently degrades to its weaker candidateSources path. (3) No production graph-quality signal: the only real precision/recall eval (bCubed/omega/labeledPairAccuracy) runs test-only against a synthetic FocusEvalPack fixture, the drift monitor only does change-detection on counts, and user feedback is a serve-time topic overlay that never feeds back to suppress a false-friend edge. Threshold governance is scattered: four different cosine constants (0.85/0.90) live in four files with no single source of truth, and the eval harness scores leiden-cpm at a different threshold (0.85/gamma-0.18) than what actually serves (0.90). Producer duplication is mostly disciplined (skip-gates, a producer selector), though CPM scoring is copy-pasted between leidenCpm.ts and topicAlgorithmComparison.ts.

**Strengths:**
- Strong edge provenance model: every ConnectionEdge carries producedBy {source, revisionId/dot/recordId} plus a confidence tier (asserted/observed/inferred) and semantic family, so edge origin is always traceable (connections/types.ts:141-231). edgeSemantics.ts additionally classifies pairwise-relatedness roles.
- The candidate-generator interface (ranker/candidates.ts) is genuinely clean: 14 named CandidateSource generators behind a uniform SourceGenerator signature, each memoized per-context via WeakMap indexes (indexesByContext) so a full generateCandidates pass is O(graph) once rather than O(visits x events) — a real CPU-regime win.
- Topic producers are skip-gated by a revisionId that is a pure function of inputs (visitSimilarity.revisionId + threshold + algorithm): on an unchanged-input drain neither the served leiden-cpm nor the shadow idf-rkn-split recomputes clustering (connectionsMaterializer.ts:3481-3610). This directly addresses the historical per-drain re-cluster CPU runaway.
- A single producer selector (servedTopicProducer.ts) with an env kill-switch and documented rollback gates which clustering serves — instant revert by restart, no code change.
- userAssertedRelations.ts embodies a correct data-quality principle: only positive user_asserted/thread attributions seed topic topology; inferred edges are explicitly excluded to avoid a low-confidence-guesses-become-topology feedback loop (documented at userAssertedRelations.ts:12-20).
- The SQLite snapshot store is reconciled to the current snapshot (delete rows absent from the snapshot, snapshot.ts:5101-5129), so stale edges DO get GC'd once the reducer stops emitting them — the persistence layer itself is not an accumulate-forever store.
- The aggregator false-friend fix is layered defense-in-depth across generation (candidates.ts COARSE_MULTI_TOPIC_DOMAINS + community keys), resolution (similarity.ts chrome-only skip), and policy (policy.ts similarity-dominant corroboration requirement) — the right architecture even where one layer is currently inert.

**Weaknesses (verified):**
- **[high] No staleness/TTL/re-validation model for inferred edges — bad title-only inference persists until the endpoint is re-observed** (`packages/sidetrack-companion/src/sync/contract/connectionsMaterializer.ts:1561`)
  Incremental similarity retains prior edges via retainedSimilarityEdgesFromSnapshot for any pair whose endpoints are not in touchedVisitIds; a visit is only re-embedded if it lands in reconcileVisitIds, which is driven by pendingTimelineVisitIds (new BROWSER_TIMELINE_OBSERVED events). A chrome-only / title-only resemblance edge (the known 1643-edge false-friend class) therefore survives indefinitely across drains. There is no TTL, no staleness marker, and no re-validation trigger keyed on edge age or evidence quality. This is the root data-lifecycle gap in the N=1 regime where a wrong edge, once formed, is effectively permanent.
- **[high] Content-vector arrival does not re-validate existing similarity edges — PAGE_EVIDENCE_EXTRACTED is content-lane-only, never re-embeds the connections visit** (`packages/sidetrack-companion/src/sync/contract/connectionsMaterializer.ts:579`)
  When better evidence (a content vector) arrives for a page that already has a title-only HNSW embedding and chrome-only edges, PAGE_EVIDENCE_EXTRACTED is a member of CONTENT_LANE_ONLY_HANDLES, not the graph HANDLES set. It invalidates the page-evidence cache (4900-4905) and marks a dirty source for a 'future content-lane reconciler worker' (789) that drains the recall search index, not the connections visit-similarity HNSW store. So the exact scenario the prompt asks about — bad inference re-validated when content vectors arrive — does not happen. corpusForVisitEntry does fold evidence into the embedding text, but only fires on re-embed, which evidence arrival never triggers.
- **[high] The served similarity edges carry no channels metadata, so the aggregator chrome-only guard is inert on them** (`packages/sidetrack-companion/src/connections/snapshot.ts:2993`)
  When persistentHnswSimilarityMode is on (the default per prior sessions), served visit_resembles_visit edges are produced by buildHnswVisitSimilarity as {fromVisitKey,toVisitKey,cosine} with no .metadata; the reducer writes only {cosine,threshold} into edge metadata (no channels). The tabsession/similarity.ts guard isChromeOnlySimilarityEdge keys off metadata.channels.contentVector (or candidateSources); with channels undefined and candidateSources absent it returns false (similarity.ts:96), i.e. HNSW edges are treated as NOT chrome-only, so the guard does not fire on the actually-served edges. The rich channel metadata (contentVector/metadata/behavior) is only produced by buildVisitSimilarity, which runs solely as an HNSW-error fallback or in legacy mode.
- **[high] No production graph-quality metric for real edges — the only precision/recall eval is test-only against a synthetic fixture** (`packages/sidetrack-companion/src/connections/focusEvalPack.ts:27`)
  runTopicAlgorithmComparison computes real quality metrics (bCubedPrecision/recall/F1, omegaIndex, labeledPairAccuracy) but is invoked only from tests, always against buildFocusEvalPack — a hardcoded synthetic set of 5 clusters at eval.sidetrack.local with hand-written pair labels. It never runs against the live vault graph. The drift monitor only does change-detection on counts (drift/driftMonitor.ts), and the feedback overlay is a serve-time filter (below). So on the real graph, edge/topic quality is discovered only via user-reported false-friends — there is no sampled precision, no ground-truth harness, no regression signal on the served graph.
- **[medium] User feedback is a serve-time topic overlay only; there is no negative-feedback channel to suppress or down-weight a false-friend similarity edge** (`packages/sidetrack-companion/src/connections/feedbackOverlay.ts:248`)
  applyFeedbackOverlayToSnapshot handles topic renames, engagement relabels, and visit_in_topic ignore/move — but it is applied on top of the snapshot at serve time and only filters visit_in_topic edges; it never touches visit_resembles_visit or closest_visit edges, and it does not feed back to suppress the underlying inference or emit a negative training signal for the pair. Combined with userAssertedRelations being positive-only, the user has no way to say 'these two pages are NOT related' and have that stick — the same false-friend edge re-surfaces every drain. In an N=1 system where hand-labeled feedback is the gold signal, discarding negative edge feedback is a material loss.
- **[medium] Threshold governance is scattered — four cosine constants across four files, with the eval harness scoring a different threshold than what serves** (`packages/sidetrack-companion/src/connections/topicAlgorithmComparison.ts:54`)
  There is no central threshold/config module for connections. Served visit-similarity emits at 0.85 (visitSimilarity.ts:63 VISIT_SIMILARITY_DEFAULT_THRESHOLD), the served leiden-cpm topic producer clusters at 0.90 (leidenCpmTopicRevision.ts:25), the shadow idf-rkn-split at 0.85 (topic-revision.ts:7), and the offline algorithm-comparison harness at 0.85 with CPM_GAMMA 0.18 (topicAlgorithmComparison.ts:54-55). So the eval that is supposed to validate leiden-cpm scores it at 0.85, but production serves 0.90 — the benchmark does not measure the served configuration. Attribution edge scores are also hardcoded inline (similarity.ts:114-125). Tuning any threshold requires editing multiple files and risks silent divergence.
- **[low] CPM scoring logic is copy-pasted between the served clusterer and the eval harness — divergence risk** (`packages/sidetrack-companion/src/connections/topicAlgorithmComparison.ts:140`)
  The leiden-CPM objective (score += internal - (CPM_GAMMA * members*(members-1))/2) is duplicated verbatim in topicAlgorithmComparison.ts:140 and leidenCpm.ts:83, each with its own local CPM_GAMMA constant. The comparison harness thus does not exercise the same code path the served producer uses; a change to the served partition math would not be reflected in the eval, and vice versa. This is the one clear producer copy-paste divergence (most other producers correctly share buildTopicRevision / the selector).

**Re-arch opportunities:**
- _freeze-safe, effort M, impact high_ — **Add an edge provenance/staleness stamp and an evidence-quality field to every inferred edge**: Stamp each visit_resembles_visit / closest_visit edge with the evidence tier that produced it (title_only | metadata | content_vector) and the producedAt of the endpoint embeddings. This is pure logging on the emit path (snapshot.ts Pass 7 already merges similarityEdge.metadata) and is prerequisite for any TTL, re-validation, or precision-sampling work. It also immediately fixes the inert aggregator guard: once the served HNSW edges carry an evidence tier, isChromeOnlySimilarityEdge can key off it instead of the never-present channels field. Freeze-safe because it is feature LOGGING, not serving-math — thresholds and which edges serve are unchanged.
- _freeze-safe, effort L, impact high_ — **Wire a content-vector arrival to re-embed the connections visit and recompute its edges**: Extend the content-lane reconciler (or add a connections-specific reconcile-visit-ids source) so that PAGE_EVIDENCE_EXTRACTED for a canonicalUrl adds that visit to reconcileVisitIds on the next drain, re-embedding it with the now-available content corpus and re-deriving its similarity edges. This is the missing 'better evidence revalidates bad inference' loop. It touches the reconcile-set plumbing, not the scoring math, so it is freeze-safe (same threshold, same producer). Effort is L because it must respect the CPU-runaway constraints — batch and bound the re-embeds, run on the worker lane, not the main loop.
- _POST-LIFT, effort L, impact high_ — **Build a real graph-quality harness over the live vault: periodic precision sampling + a negative-feedback edge channel**: Two parts. (a) A sampling job that pulls K served similarity/topic edges per week and records them for a lightweight in-product thumbs-up/down, producing a real precision estimate on the user's own graph instead of the synthetic FocusEvalPack. (b) A USER_REJECTED_RELATION event that, unlike the current serve-time topic overlay, is persisted and consumed as a suppression on the specific pair (and as a negative label for the ranker). This closes the 'quality only discovered via user-reported false-friends' gap and gives the N=1 system the hand-labeled gold it is starved for. Data-collection + logging + eval-infra are explicitly freeze-safe; the suppression application to served edges is a serving-math change and must be gated behind the freeze until the demo passes.
- _freeze-safe, effort M, impact medium_ — **Centralize connections thresholds in one config module and make the eval harness read the served config**: Create connections/thresholds.ts as the single source for the similarity emission threshold, the served topic cosine + gamma, and the attribution source scores, then have leidenCpmTopicRevision.ts, visitSimilarity.ts, and topicAlgorithmComparison.ts import from it. Point runTopicAlgorithmComparison at the served threshold (0.90) so the benchmark measures the shipped configuration, and dedupe the CPM scoring so the harness calls leidenCpm.ts rather than its own copy. Pure refactor with no behavior change if the extracted constants preserve current values — freeze-safe as a storage/layout change. Effort S-M.

### data-platform

The data platform is a JSONL-canonical, event-sourced store (per-replica append-only `_BAC/log/<replica>/<date>.jsonl`) with a well-engineered read layer bolted on over several CPU-runaway incidents: a signature-memoized `readMerged` with 60s idle eviction, append-path indexes that removed the per-write full-log rescan, watermark-resume tail reads (`readMergedSince`), and type-pre-filtered streaming (`streamFiltered`/`streamEvents`) that skips JSON.parse on the ~90% engagement.interval bulk. Derived state is demand-paged and idle-evictable (the connections snapshot lives in SQLite `current.db` with a 60s-idle in-RAM cache; the semantic pool rebuilds incrementally), which is exactly right for a laptop-resident recsys and explains why the memory picture is manageable despite the 2.4-3.6GB floor. The core weakness is proliferation without lifecycle: I count ~9 distinct on-disk formats/engines (JSONL log, projection-changes.jsonl, per-aggregate projection JSON, and four+ SQLite DBs — event-store, engagement-facts, timeline-facts, connections current.db, recall v2 index) plus a semantic vector store, and the two canonical append-only logs (the event log and projection-changes.jsonl) have NO compaction/retention path — the GC plan only prunes derived revision files, never the logs. The optional SIDETRACK_EVENT_STORE is default-OFF for a measured-real reason (raw-row fetch doesn't fix the JS heap and costs CPU on hot polls), which means the platform maintains BOTH a JSONL-scan path and a store path in every hot consumer — double code, dual correctness burden, and the store's real payoff (SQL aggregation, not raw rows) is unbuilt. For N=1 the single-replica (replicaId,seq) ordering is sound and full-history retrains in seconds are the right call; the sustainability risk is the forever-growing engagement.interval log under the still-default JSONL training scan.

**Strengths:**
- readMerged is memoized on a cheap filesystem signature (shard mtime+size) with in-flight dedupe and a 60s idle-TTL sweep that drops the hundreds-of-MB event array during idle stretches — the single most important fix behind the historical CPU runaways (sync/eventLog.ts:653-757).
- Append-path indexes (clientEventId->dot, dot-key set, aggregate->deps vector) removed the per-write full-log readMerged that caused 46-69s POSTs; appends are now O(batch)+one file append and no longer re-warm the memo (sync/eventLog.ts:906-932).
- Type-aware streaming reads avoid materializing the full merged array: streamEvents/streamFiltered pre-test the raw line for `"type":"<t>"` before JSON.parse, so the ~88-92% engagement.interval bulk is never parsed for the small-subset consumers (sync/eventLog.ts:810-872).
- The connections graph snapshot is demand-paged from SQLite (SqliteConnectionsStore) with a revision-keyed in-RAM cache that idle-evicts after 60s — the ~150MB snapshot is NOT permanently pinned; the next reader pays a sub-100ms paged read (connections/snapshot.ts:4047-4099). This is the correct pattern the raw event store failed to achieve.
- Watermark-resume drain reads: readMergedSince advances the connections materializer's frontier by reading only past-frontier shard tails, short-circuiting whole shards whose tail seq is already covered, independent of the readMerged memo (sync/eventLog.ts:759-801).
- Store and JSONL paths share ONE ordering — both sort/scan by (replicaId, seq) — so the dual read paths return byte-identical order and last-write-wins folds are unaffected (sync/causal.ts:172-177, eventStore.ts:424-448).
- Torn-tail durability is real: parseLine treats a non-parsing final line as a data-loss signal (not silent), readShardTailSeq scans backward to recover the last valid seq so a crash mid-append can't reissue a duplicate dot, and ShardUnreadableError distinguishes EACCES/EIO from ENOENT so a transient per-shard failure skips-and-retries instead of advancing the frontier past unread events (sync/eventLog.ts:316-334, 490-520, 778-796).
- The event store's incremental catch-up is genuinely incremental: shard_progress tracks per-file (size, mtime, read_offset) and only reads the byte tail past the last offset, chunked at 2000 with cooperative yields (sync/eventStore.ts:311-413).

**Weaknesses (verified):**
- **[high] The canonical JSONL event log has no compaction or retention — it grows forever, and the GC plan explicitly excludes it** (`packages/sidetrack-companion/src/gc/plan.ts:154`)
  buildGcPlan prunes only DERIVED artifacts (visit-similarity/topic/closest-visit revisions, diagnostics, debug-dumps, idempotency receipts). The append-only JSONL log under _BAC/log/ is never a GcGroup. Since the log is ~88-92% engagement.interval (per the code comments) and every nav/engagement tick appends, the log grows monotonically with browsing, and every full-log consumer (readMerged, the default training scan) scales with total lifetime history, not with the useful signal. There is no rollup of old engagement.interval into engagement.session.aggregated + drop.
- **[high] Full training-history scan on every retrain falls back to a whole-log streamFiltered when the event store is OFF (the default)** (`packages/sidetrack-companion/src/sync/contract/connectionsMaterializer.ts:4154`)
  readTrainingEvents uses the store's typed index (forEachChunkOfTypes, ms-scale) ONLY when storeBackedEvents !== null. eventStoreEnabled() is default-OFF, so the production path is deps.eventLog.streamFiltered over RANKER_TRAINING_EVENT_TYPES — a full byte-scan of every shard each retrain. The type-needle pre-filter skips JSON.parse on non-matching lines but still reads 100% of the log bytes off disk. As the engagement.interval bulk grows forever (weakness above), this read cost grows unboundedly even though the trainable subset (recall.served/action + feedback) is tiny and grows only at human speed. This is the read-amplification twin of the retrain-starvation bug already fixed once.
- **[medium] projection-changes.jsonl is a second unbounded append-only log, fully re-read and re-parsed on every browser sync poll** (`packages/sidetrack-companion/src/sync/projectionChanges.ts:177`)
  readSince reads the ENTIRE projection-changes.jsonl with readFile + splits + JSON.parses every line, filtering to seq > sinceSeq, on every /changes poll (e.g. /v1/review-drafts/changes). The file self-documents 'pruning is left as a future optimisation' and grows one line per projection change forever. On an active vault with frequent thread/annotation/dispatch edits this becomes an O(total-history) read on a hot polled endpoint — the exact readMerged anti-pattern, in a different file, without the memo.
- **[medium] Storage-engine sprawl: ~9 distinct on-disk formats with an implicit derivation DAG and no single documented lineage/rebuild contract** (`packages/sidetrack-companion/src/sync/eventStore.ts:50`)
  Persistent stores now include: JSONL log (canonical), projection-changes.jsonl, per-aggregate projection JSON, event-store.db (default OFF), engagement-facts.db (derived), timeline-facts.db (default OFF), connections current.db + daily snapshot JSON, recall/v2/index.sqlite, and semantic-pool vectors.json + recall/index.bin. Each is independently rebuilt-from-JSONL with its own catch-up/watermark logic (three near-identical latestPerAggregateFromLog implementations in reproject.ts, antiEntropy.ts, and the materializer). The DAG (JSONL -> {event-store, engagement-facts, timeline-facts} -> snapshot -> {topics, ranker, recall vectors}) is real but only encoded in scattered comments; there is no one module that owns 'which store derives from what, and how to rebuild it'. This is high per-store code-and-bug surface for a single-user system.
- **[medium] Maintaining both the JSONL-scan path and the event-store path in every hot consumer is the worst-of-both-worlds until the store earns its keep via aggregation** (`packages/sidetrack-companion/src/sync/antiEntropy.ts:62`)
  Because SIDETRACK_EVENT_STORE is default-OFF, every hot consumer carries a `store === null ? readMerged/streamFiltered : store.forEachChunk...` fork: antiEntropy, reproject, workGraphHealth, recall/ingestor, urls/autoApply, tabsession/autoApply, and the materializer's training read. That doubles the branches to test and keeps two correctness invariants live per consumer. The store's own comment states the raw-row fetch is net-negative and the real win is SQL AGGREGATION (rolled-up URLs/sessions/metrics), which is not built for these consumers — so today the fork buys nothing on the default path and is pure maintenance tax.
- **[low] Daily connections snapshots accumulate on disk with no GC** (`packages/sidetrack-companion/src/connections/snapshot.ts:3603`)
  The connections store writes 'rolling current.json + daily snapshots' under _BAC/connections/snapshots/. The GC plan prunes visit-similarity/topic/closest-visit/diagnostics revisions but has no snapshots group, so daily snapshot files accumulate one-per-day indefinitely. Lower severity than the log because each is a bounded single-day artifact, but it is another un-swept growth surface on a laptop.
- **[low] Periodic anti-entropy re-warms the full-log memo every 30 min even when the extension is idle (store default-OFF)** (`packages/sidetrack-companion/src/sync/antiEntropy.ts:63`)
  scanOnce runs every 30 min and, on the default store-OFF path, calls eventLog.readMerged() to pick latestPerAggregate — re-materializing the whole event array (hundreds of MB) and defeating the 60s idle-TTL that would otherwise let the memo drop during idle stretches. It then runs runImportProjectors per aggregate, each calling readByAggregate -> readMerged (memo is warm by then, so O(filter), not O(read)), but the initial re-warm alone pins memory and burns CPU on a laptop that may have been idle for hours.

Trivial, mirrors the existing oldRevisions helper, freeze-safe (derived-file lifecycle only). Closes weakness #6 and gives a single home for future log-archive retention once compaction lands.

## Freeze-safe roadmap (executable now, ordered)

### 1. Persist point-in-time served features + query-anchored cosine into recall.served  _(effort M, impact high)_

Add two fields to RecallServedCandidateSnapshot (recall/events.ts): the exact extractFeatures vector and the per-candidate query-to-candidate cosine already computed in the pipeline, plus keep the existing servedPosition. Today the snapshot logs perLaneRanks/fusedScore/rerankScore/servedPosition but NOT the feature vector or the cosine, so the trainer re-derives features at retrain time via extractFeatures(candidate,{merged:[...merged],snapshot}) against a drifted graph, and the single most-relevant signal (query cosine) is thrown away. This is a clean field-addition to an already-logged payload — no new event, no serving-math change, no threshold change.

**Why now:** THE keystone, on which all three stances converge and everything downstream depends: it kills train/serve feature skew and the referenceMs future-leakage class AT THE SOURCE (before it becomes harmful once serving math changes post-lift), revives the dead semantic signal as a first-class logged feature, resolves the anchorId query-only train/serve mismatch, captures servedPosition so it can later become an IPW term instead of a raw feature, and is the prerequisite for a true offline replay harness. Time-critical: every day not logging is a day of unrecoverable as-of serve context — features cannot be reconstructed after the graph drifts.

### 2. Log shown-but-unclicked as weak negatives + open a persisted USER_REJECTED_RELATION channel  _(effort M, impact high)_

Two label-density fixes. (a) In buildRecallImpressionTrainingGroups, stop discarding unjudged shown candidates: emit them as weak/hard-negative rows (reuse the dormant negatives.ts samplers that today only run in the legacy path) so a group is no longer one-positive-plus-explicit-rejects. (b) Add a persisted USER_REJECTED_RELATION event so the user can say 'these two pages are NOT related' — COLLECT and STORE it now; applying it as edge suppression is deferred behind the freeze. Both are collection/build changes, not serving-math.

**Why now:** The N=1 system is starved of its only gold signal, yet discards negatives twice (feedbackOverlay is serve-time-only and touches only visit_in_topic; impression training drops unjudged rows). This is the densest label-yield increase available and it must be LIVE before the demo so the demo window itself accrues labels — a rejection a user makes during the freeze cannot be collected retroactively. Feeds ranker, graph-quality harness, and eventual edge suppression from one channel.

### 3. Index the trainable subset into a dedicated store + add cooldown + move the train off-thread  _(effort M, impact high)_

Three coupled infra fixes. (a) Maintain a tiny append-only trainable-events shard (or one type-indexed SQLite table) written at emit time for the ~6 low-volume trainable types (recall.served/action + 4 feedback) so retrain reads O(labels), independent of the default-OFF SIDETRACK_EVENT_STORE — removing the verified whole-log streamFiltered byte-scan at connectionsMaterializer.ts:4154. (b) Extend the planRankerRetrain cooldown/fingerprint guard (today only on the legacy path, 326-372) to the impression path (890-956) so a no-new-label drain short-circuits. (c) Run train() off the drain thread via the already-existing trainGroups.worker.ts / runMaybeRetrainInWorker. No scoring change, no label change.

**Why now:** The inline per-drain retrain with no cooldown is the exact re-embed/rebuild shape behind the historical 100-144% CPU runaways — and the plumbing to fix it already exists. The whole-log training scan is the read-amplification twin of the already-fixed retrain-starvation bug; it degrades silently as the forever-growing engagement.interval bulk accumulates. Sequence after move 1 defines the enriched schema so the store captures the right shape once (avoid a second migration). This is CPU-regime hygiene that must land before label-density work increases retrain frequency.

### 4. Stamp evidence-tier provenance on served vectors and inferred edges; thread body_indexed through queryVector  _(effort S, impact high)_

Two symmetric emit-path/read-path additions. (a) queryVector (recall-v2/store/sqlite.ts:560) SELECTs only entityId/canonicalUrl/title/cosineDistance — add the existing body_indexed column (it is in the docs schema but never returned) so serving can distinguish title-only from content vectors. (b) Stamp each visit_resembles_visit / closest_visit edge with its evidence tier (title_only|metadata|content_vector) plus endpoint embedding producedAt; the reducer already merges similarityEdge.metadata at snapshot.ts:2993. Pure logging/read-column — thresholds and which edges serve are unchanged.

**Why now:** Cheap (S), independent, and it immediately makes the currently-INERT aggregator chrome-only guard functional: isChromeOnlySimilarityEdge keys off metadata.channels.contentVector, which the served HNSW edges never carry, so it silently returns false on the actually-served edges — the evidence tier gives it something real to read. It is also the prerequisite for any TTL/re-validation and for provenance-aware precision sampling (move 6). Merged from the content-pipeline and graph-product analyses; the strongest formulation credits both the queryVector column gap and the edge-metadata gap.

### 5. Codify the derivation DAG in one lineage module + add lifecycle to growth surfaces (planner only)  _(effort M, impact medium)_

(a) Create one lineage/registry module declaring every derived store (event-store.db, engagement-facts.db, timeline-facts.db, connections current.db + daily snapshots, recall v2 index.sqlite, semantic-pool vectors.json), its source event types, rebuild entrypoint, and default state; collapse the three latestPerAggregateFromLog copies (reproject.ts, antiEntropy.ts, materializer) into one. (b) Ship the SAFE half of compaction: a report-only planner that reports reclaimable engagement.interval bytes per sealed past-day shard (mirrors gcInventory) WITHOUT deleting; extend GcGroup with bounded-retention for _BAC/connections/snapshots/ (keep newest N) and add offset-cursor resume + pruning to projectionChanges.readSince (today it readFiles+JSON.parses the ENTIRE file on every /changes poll). Destructive log deletion is NOT freeze-safe (append indexes reject in-process shard rewrites; dropping events changes projection folds) — planner now, deletion post-lift.

**Why now:** This is the cheap map that de-risks every storage move and pays down the dual-path maintenance tax; doing compaction before a single lineage owner means editing rebuild logic in three places. The report-only planner sizes the forever-growth win without touching serving-adjacent folds, the derived-file GC and projection-changes cursor close real O(history) reads on hot paths, and the lineage module gives the eventual event-store default-ON decision one home. Lower urgency than the ceiling-raising moves (it protects the floor, not the ceiling) so it runs in the background.

### 6. Stand up freeze-safe eval infra: replay harness + scored connections precision + perf-regression lane  _(effort L, impact high)_

Build the measurement layer, all freeze-safe. (a) A served-feature replay driver that reads the point-in-time rows from move 1 to replay real logged impressions through candidate models + the deterministic baseline, computing nDCG@10/MRR/recall@k AND explicit-reject false-positive-rate on the user's real vault (not synthetic FocusEvalPack). (b) A connections false-merge/precision harness fed from buildWorkGraphEvalAcceptedEvents + the move-4 evidence tiers, run against the LIVE graph, plus a weekly sampling job that pulls K served edges for lightweight in-product thumbs-up/down. (c) A perf-regression lane: per-drain and per-endpoint time budgets asserted at realistic vault scale (unit scale hid the 6.8s leiden blowup), a growth inventory flagging any hot read that scales with total history, and a no-new-input-drain-recomputes-nothing assertion. (d) Ready-to-swap gate math: paired per-group bootstrap significance + minimum-effect threshold + reject-at-top-1 fp-rate floor (built, not yet flipped into promotion).

**Why now:** Below the ranker path there is no scored offline eval and no production graph-quality signal; the gate promotes on epsilon deltas. This converts 'we think the model is better' into 'we can prove it on this user's data' — which is precisely the gate for lifting the freeze. It consumes moves 1/3/4, so it follows them, and it must be battle-tested on accumulated data by the time the demo passes, so the post-lift serving changes have a real harness the instant they are legal. The perf-regression lane is the ratchet that keeps moves 3 and 5 from silently regressing into the next CPU incident.

## Post-lift roadmap (gated on §13 demo 16/16 + §15 window, in eval-supported order)

### 1. Switch the learned head to pointwise-residual-over-graph-baseline; retire the LambdaRank machinery  _(effort L, impact high)_

Replace the listwise/pairwise LambdaRank objective with a pointwise objective that treats the deterministic graph_baseline score as a fixed prior and learns a small, regularized residual. LambdaRank is statistically weak at one-positive-per-impression, and the very property that made the referenceMs offset harmless (within-group relative ranking) is a sign it is fighting to learn structure the data cannot support. Rebase the clamped online RankNet head onto the same residual base (its rebaseToModel design is already correct).

**Why:** The strong-prior + tiny-learned-correction pattern is the small-data-ML move, versus the fleet move (large-capacity listwise + volume). This is the payoff, but it changes serving math and is unshippable without the move-6 replay harness to prove it beats the deterministic baseline on the user's real impressions. Sequence first among post-lift because everything else (debiasing, thresholds) tunes around the new objective.

### 2. Drop served_position as a raw feature; replace with an IPW term over logged propensity  _(effort M, impact high)_

Remove servedPosition as a trained input and, using the propensity logged by an epsilon-greedy exploration slot, apply inverse-propensity weighting to reconcile exploration and organic impressions. The exploration slot's selection propensity is logged during the freeze (piggybacks on move 1); only the flip that puts the exploration candidate into the served order is a serving change and waits for this window.

**Why:** servedPosition hard-codes 'top slot = relevant' with no fleet averaging to wash out the position-bias feedback loop, and with zero exploration the model only ever sees what it already ranked highly. IPW is the correct debiasing even in industry and is mandatory here. Depends on the logged propensity from the exploration-slot plumbing landed pre-lift, and on the move-6 harness to verify the debiased model does not regress.

### 3. Unify the ship-gate positive floor to one env-threaded N-for-N=1 number, with k-fold when groups are scarce  _(effort S, impact high)_

Thread ONE env-tunable floor through both the trainer (minRecallImpressionPositiveGroups) and the gate (which today omits minImpressionsWithPositive and falls to DEFAULT=50, a fleet number). When reserved groups are too scarce for a clean temporal holdout, evaluate on a temporal k-fold instead of a single 20% slice so the gate renders a real verdict earlier without leaking. The plumbing unification and k-fold mechanism are freeze-safe infra; the actual floor VALUE is a promotion-threshold change that must be set here, informed by what the move-6 harness shows about the user's real positive-group cadence.

**Why:** 50 is a fleet number in an N=1 world and the two floors are unplumbed from each other. Set it too low and you promote noise; too high and the model is permanently cold-started — the exact N=1 trap. The number can only be chosen once the replay harness has measured real label cadence, so it lands in this window even though the plumbing/k-fold half could ship earlier.

### 4. Serve from documents_chunks_vec with max-chunk pooling; down-weight title-only hits  _(effort M, impact high)_

Add queryChunkVector KNN + doc-level max-chunk pooling and prefer it over docs_vec where clean chunk vectors exist; use the body_indexed / evidence-tier provenance (threaded pre-lift) to down-weight title-only vectors in fusion. backfillChunkVectors already embeds real chunk body text into documents_chunks_vec, but the pipeline queries only docs_vec — the clean signal is embedded and never read.

**Why:** ~86% of docs_vec is title-only (13.6% content coverage), the direct cause of aggregator false-friends in recall ranking, while the good chunk vectors sit unused. This directly raises retrieval precision, but flipping which vectors serve is a serving-math change gated behind the freeze and must be validated on the move-6 recall harness (enriched to 384-dim + a hand-labeled replay set).

### 5. Wire content-vector arrival to re-embed the connections visit; add TTL/re-validation + edge suppression  _(effort L, impact high)_

Extend the reconcile-set so PAGE_EVIDENCE_EXTRACTED for a canonicalUrl adds that visit to reconcileVisitIds on the next drain, re-embedding it with the now-available content corpus and re-deriving its similarity edges — the missing 'better evidence revalidates bad inference' loop (today PAGE_EVIDENCE_EXTRACTED is CONTENT_LANE_ONLY and never touches the visit-similarity HNSW). Add a TTL/staleness model so a title-only chrome-only resemblance edge does not persist indefinitely, and apply the persisted USER_REJECTED_RELATION (collected pre-lift) as a hard suppression on the specific pair + a negative ranker label.

**Why:** In N=1 a wrong edge, once formed, is effectively permanent (the 1643-edge false-friend class) because edges are only recomputed when the endpoint is re-observed. Closing this loop is high-value but it changes which edges serve, must respect the CPU-runaway constraints (batch/bound re-embeds on the worker lane, not the main loop), and needs the move-6 graph-quality harness to confirm it improves precision without collateral suppression. Sequence last: highest blast radius, depends on the evidence tiers, the reject channel, and the harness all being in place.

## Anti-patterns (industry-standard moves that are wrong in this regime)

- Two-tower / dual-encoder learned retrieval. Standard fleet candidate-generation, pointless at N=1: there are only thousands of vault items, exact-kNN over sqlite-vec is already cheap and correct, and a learned tower needs millions of interaction pairs it will never see — the towers never leave random init at tens of labels/day. Keep exact retrieval + RRF + a small learned reranker; do NOT learn retrieval.
- Fleet-style feature store (Feast-style materialized online/offline tables) as the training-feature source of truth. It inverts the correct fix: the problem is not that features aren't centrally stored, it's that they aren't captured POINT-IN-TIME at serve. Logging the served feature row into the recall.served payload gives exact as-of features with zero new infra; a feature store re-derives from the drifted graph and REINTRODUCES the leakage, while adding a 10th on-disk format to an already-sprawling layout.
- Nightly / distributed / scheduled batch training on a cluster. Full-history retrain finishes in seconds at N=1, so a batch pipeline adds a scheduler, feature store, and backfill jobs for zero latency benefit — and a wall-clock cron fires against a laptop process that may be asleep. The on-device-correct equivalent is drain-triggered, worker-offloaded, cooldown-guarded retrain reading an O(labels) shard. The pathology to fix is per-drain no-cooldown retrain, not the absence of a scheduler.
- High-capacity listwise LambdaMART as the primary objective. The reflex ranking choice, but listwise/LambdaRank loss is statistically hopeless at one-positive-per-impression; its robustness to the referenceMs offset comes precisely from within-group relative ranking, a tell that it cannot learn absolute structure from this data. Use a strong deterministic prior + a small regularized pointwise residual.
- Large randomized exploration / full contextual-bandit serving to debias position. Fleet systems spend impressions on exploration because they have millions to spare; at N=1 every served slot is one of a handful the single user sees per day, so aggressive randomization directly degrades the only user's experience and its per-arm statistics never tighten (this is why the online head is correctly default-OFF). The regime-appropriate move is a single small seeded epsilon-greedy slot with logged propensity + IPW — and even that serve-side flip waits for the window; only the logging is free now.
- Treating implicit signals (dwell, hover, click-through as pseudo-positives) as gold, mined at volume. Fleet systems drown pointwise noise in scale; here labels arrive at human speed and explicit hand-labeled feedback IS the gold. Treating noisy implicit signal as ground truth corrupts the scarce clean signal. Keep explicit-only positives/negatives; use unjudged-shown only as WEAK negatives, never pseudo-positives.
- Cross-user transfer learning / warm-start from a base model trained on other users' data — the standard cold-start remedy. Violates the privacy-first, nothing-leaves-the-device invariant that is the product's reason to exist, and there is no fleet to transfer from. Cold-start is permanent-ish BY DESIGN; the answer is maximal extraction from the single user's own gold labels (dense negatives, persisted rejections, point-in-time features), not a population prior.
- Flip SIDETRACK_EVENT_STORE default-ON to unify onto SQLite and delete the JSONL-scan fork wholesale. The store is default-OFF for a MEASURED-real reason: raw-row fetch doesn't fix the JS heap and costs CPU on hot polls; its real payoff (SQL aggregation) is unbuilt for these consumers. Flipping it today pins memory and burns CPU on exactly the hot paths this regime can't afford, for zero aggregation benefit. Build SQL aggregation for 1-2 genuinely-aggregate consumers (health/diagnostics counts) first, prove the win, THEN retire those forks.
- Aggressive event-log deletion / retention TTL now to cap growth. Dropping raw events is genuinely load-bearing but is NOT freeze-safe: which events fold into projections/graph is serving-adjacent, and the append-path indexes explicitly do not tolerate in-process shard rewrites. The 'just add a TTL' reflex would silently change fold results. Ship a report-only compaction planner now; gate the destructive rewrite behind the section-15 window, operating only on sealed past-day shards whose intervals are already aggregated.
- Scaling up the model (deep cross-encoder, LLM reranker, larger embedding model) to raise quality. Bigger models raise the CPU/memory floor — the dominant historical failure mode (100-144% CPU, 2.4-3.6GB floor) — while doing nothing about the binding constraint, which is label + coverage starvation. A larger model over the same 13.6%-coverage title-only vectors just learns the same false-friends faster. Fix the data floor first; LightGBM LambdaRank is not the bottleneck.

## North star (6-month target architecture)

In six months Sidetrack is a point-in-time-honest, self-measuring single-user recsys: every served impression durably records its own as-of feature vector, query-anchored cosine, served rank, and selection propensity, so the trainer replays real history instead of re-deriving features against a drifted graph, and every shown-but-unclicked candidate plus every user 'not related' rejection is a first-class label — turning tens-of-labels-per-day into the densest possible signal. Serving stays deliberately simple and on-device: exact-kNN over the vault (chunk-vector-pooled, provenance-tagged so title-only hits are down-weighted), RRF, a cross-encoder, and a small pointwise residual learned over the deterministic graph_baseline with served_position debiased via inverse-propensity weighting, promoted only through a paired-bootstrap significance gate with a single env-threaded N=1 floor and a reject-precision floor, validated on a scored replay harness over the user's own vault. The graph carries evidence-tier provenance on every inferred edge, re-validates bad inference when content vectors arrive, expires stale title-only edges, and honors persisted negative feedback — with a live precision-sampling loop replacing the synthetic fixture. Underneath, the derivation DAG is codified in one lineage module; the trainable subset lives in a tiny O(labels) store read independent of the main event store; all heavy compute (embed, train, partition) runs off the drain thread on bounded worker lanes behind cooldown/fingerprint guards; the forever-growing JSONL log is compacted at sealed past-day shard boundaries with all growth surfaces under GC; and a standing perf-regression lane asserts per-drain time budgets at real vault scale so the next CPU runaway fails CI instead of the user's laptop. Every fleet-scale reflex — feature stores, two-tower retrieval, nightly distributed training, cross-user transfer, aggressive exploration — is deliberately absent, because the winning moves in this regime are the ones industry rarely needs: full-history retrain in seconds, exact-kNN over thousands of items, and hand-labeled feedback treated as gold.
